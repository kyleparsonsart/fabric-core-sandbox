Overview

Why spacing matters in Apollo

Apollo is a data-dense sales platform — users are scanning hundreds of records, switching between workflows, and making time-sensitive decisions in a single session. Spacing is the primary signal that tells users where one thing ends and another begins. Each value in the four-step scale serves a distinct structural role — from separating major page regions down to binding sub-elements within a single component. Spacing decisions should be based on the relationship between elements, not visual preference.

Scannability
Predictable rhythm lets users parse tables and lists faster.
Clarity
Consistent gaps communicate which elements belong together and which don't.
Reduced friction
Designers and engineers stop debating spacing case-by-case.
Maintainability
A token-based system means spacing can be updated globally from a single source.
Spacing Scale

Four values, two roles

Spacing values serve two distinct roles: gap (space between elements) and padding (space inside a container). The same value can serve both roles depending on context.

Gaps between regions
gap/region · 24px
The sidebar and main content are each a region. A 24px gap separates them — the widest break in the layout hierarchy.
Padding within sections
padding/section · 24px or 16px
Standalone page-level cards use 24px padding. Nested or compact cards use 16px — padding follows context, not preference.
Gaps between zones
gap/zone · 16px
The tab bar, toolbar, and table are each zones within a section. A 16px gap separates them — tighter than section gaps but still clearly divisive.

Page Hierarchy

Region, Section, Zone

Apollo's layouts are composed of three levels. Understanding which level you're working at determines which spacing value to reach for. Below the zone level, you're working inside a component — a discrete UI element like a table row, a chip, or a form field.

01
Region
gap/region · 24px
The highest-level division of a page. The app sidebar and the main content area are each a region. Regions are always separated by a 24px gap.
02

A self-contained block of content within a region. "Suggested leads" and "Email performance funnel" are each a section, separated by a 24px gap.

Principles

How to apply this system

Six rules that should govern every spacing decision made in Apollo's UI.

01
Gap and padding are different roles, not different values.
24px means something different as card padding than it does as section gap. Always consider which role a value is serving before applying it.
02
Card padding follows context, not preference.
A standalone page-level card gets 24px padding. A nested or compact card gets 16px. Don't apply 24px padding to a card that lives inside another surface — it will feel disconnected from its container.
03
Space communicates relationship.
The smaller the gap, the stronger the implied connection. Use 4px only when two elements are genuinely sub-parts of the same component. Use 24px only when you intend a hard visual break.
04
Padding doesn't have to be uniform.
Applying padding to one or two sides only is common throughout Apollo — use padding-top, padding-left, etc. to be explicit. When describing asymmetric padding in redlines, always name the sides: "16px padding-left" not "16px padding". Never describe padding as shorthand when only certain sides are intended.
05
Be consistent within a surface type.
All data table surfaces should share the same spacing rhythm. One-off values erode the system.
06
Don't use spacing to compensate for missing hierarchy.
Extra padding won't make something feel like a section heading. That's a typography problem.
07
Canvas surfaces are out of scope.
Don't apply these rules to the Workflow builder or other canvas layouts. Separate documentation will follow in Phase 02.
Corner Radius

Radius communicates hierarchy

Corner radius is part of the same visual foundation as spacing and follows the same two-layer token architecture. Radius values communicate component hierarchy and surface elevation — smaller radii for nested or sub-level elements, larger radii for elevated floating surfaces.

Apollo's current usage — 8px for components and 12px for modals — aligns well with industry standards. Material Design 3 uses the same values for equivalent surface types. Fluent 2 is slightly more conservative at 4–8px, reflecting Microsoft's enterprise-formal aesthetic. Apollo's "more human" brand direction makes 8px and 12px the right call.

The Nesting Rule

Always step down for nested elements

When a component sits inside a container, the inner element's radius must always be smaller than the outer container's radius. If they're equal, the inner corners appear visually flat against the outer curve — a subtle but noticeable inconsistency.

Apollo's current values already respect this: an 8px button inside a 12px modal has appropriately nested radii. As a general rule, step down one level in the scale for each level of nesting.

Principles

Four rules for corner radius

01
Always step down for nested elements.
An inner element's radius should always be smaller than its container's. Use the nesting diagram above as the default reference — only deviate with explicit justification.
02
radius/none is intentional, not a default.
A flush element inside a rounded container is a deliberate signal that it's structural rather than interactive. Don't use 0px just because a component looks fine without rounding.
03
Don't introduce intermediate values.
Values like 6px or 10px break the nesting rule and create visual inconsistency. If a component looks too round or too sharp, reconsider its surface level rather than adding a new token.
04
radius/pill uses 9999px, not 50%.
This ensures correct rendering across browsers and platforms at any component height. 50% creates an oval at non-square dimensions — 9999px always produces a true pill.
