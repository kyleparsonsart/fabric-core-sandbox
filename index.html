Overview

Why spacing matters in Apollo

Apollo is a data-dense sales platform — users are scanning hundreds of records, switching between workflows, and making time-sensitive decisions in a single session. Spacing is the primary signal that tells users where one thing ends and another begins. Each value in the four-step scale serves a distinct structural role — from separating major page regions down to binding sub-elements within a single component. Spacing decisions should be based on the relationship between elements, not visual preference.

Scannability
Predictable rhythm lets users parse tables and lists faster.
Clarity
Consistent gaps communicate which elements belong together and which don't.
Reduced friction
Designers and engineers stop debating spacing case-by-case.
Maintainability
A token-based system means spacing can be updated globally from a single source.
Spacing Scale

Four values, two roles

Spacing values serve two distinct roles: gap (space between elements) and padding (space inside a container). The same value can serve both roles depending on context.

Gaps between regions
gap/region · 24px
The sidebar and main content are each a region. A 24px gap separates them — the widest break in the layout hierarchy.
Padding within sections
padding/section · 24px or 16px
Standalone page-level cards use 24px padding. Nested or compact cards use 16px — padding follows context, not preference.
Gaps between zones
gap/zone · 16px
The tab bar, toolbar, and table are each zones within a section. A 16px gap separates them.

Principles

How to apply this system

Six rules that should govern every spacing decision made in Apollo's UI.

01
Gap and padding are different roles, not different values.
24px means something different as card padding than it does as section gap. Always consider which role a value is serving before applying it.
02
Card padding follows context, not preference.
A standalone page-level card gets 24px padding. A nested or compact card gets 16px.
03
Space communicates relationship.
The smaller the gap, the stronger the implied connection. Use 4px only when two elements are genuinely sub-parts of the same component.
04
Padding doesn't have to be uniform.
Applying padding to one or two sides only is common throughout Apollo. When describing asymmetric padding in redlines, always name the sides.
05
Be consistent within a surface type.
All data table surfaces should share the same spacing rhythm. One-off values erode the system.
06
Don't use spacing to compensate for missing hierarchy.
Extra padding won't make something feel like a section heading. That's a typography problem.
07
Canvas surfaces are out of scope.
Don't apply these rules to the Workflow builder or other canvas layouts. Separate documentation will follow in Phase 02.
Corner Radius

Radius communicates hierarchy

Corner radius is part of the same visual foundation as spacing and follows the same two-layer token architecture. Radius values communicate component hierarchy and surface elevation — smaller radii for nested or sub-level elements, larger radii for elevated floating surfaces.

Apollo's current usage — 8px for components and 12px for modals — aligns well with industry standards. Material Design 3 uses the same values for equivalent surface types. Apollo's "more human" brand direction makes 8px and 12px the right call.

The Nesting Rule

Always step down for nested elements

When a component sits inside a container, the inner element's radius must always be smaller than the outer container's radius.

radius/surface12px
radius/component8px
radius/element4px
Principles

Four rules for corner radius

01
Always step down for nested elements.
An inner element's radius should always be smaller than its container's.
02
radius/none is intentional, not a default.
A flush element inside a rounded container is a deliberate signal that it's structural rather than interactive.
03
Don't introduce intermediate values.
Values like 6px or 10px break the nesting rule and create visual inconsistency.
04
radius/pill uses 9999px, not 50%.
This ensures correct rendering across browsers at any component height.

Color System

Primitive palette, semantic token assignments, and WCAG contrast pairings — Apollo's warm neutral foundation with a single brand accent.

Primitives · Color

11 named ramps

Apollo v3 organizes its palette into 11 named color ramps. Each ramp runs from light (10) to dark (80–90), giving semantic tokens a consistent dimensional reference. Do not consume primitive tokens directly in components — always reach for a semantic token.

Semantics · Color

Intent-encoded tokens

Semantic tokens map ramp steps to named roles across three categories: Foreground (text and icon colors — fg tokens apply to both), Background (surface fills), and Border (strokes and dividers).

Contrast Pairing Matrix

Which text/background pairs pass WCAG AA

Evaluated against WCAG 2.2 AA thresholds. Standard text requires 4.5:1; large text (≥18pt regular or ≥14pt bold) requires 3:1.

Overview

A deliberate palette

Apollo's color system is built around a single warm neutral scale — not a traditional multi-hue design system palette. This is intentional. Apollo is a data-dense platform where color competes with user-generated content: names, company logos, status chips, email threads. A neutral foundation ensures the interface recedes when it should, and the user's data takes center stage.

Apollo Yellow (the sun ramp, default #FBF500 · sun-10) is the sole expressive accent. Its use is reserved for brand-anchored moments — not status colors, not interactive states.

Scannability
Neutral backgrounds and high-contrast text let users scan dense tables and lead lists without visual fatigue.
Content first
The interface shouldn't compete with user data. Warm neutrals subordinate the chrome so leads, contacts, and metrics stand out.
Brand restraint
Apollo Yellow is expressive — use it for brand moments, not functional UI. Overuse dilutes its signal.
Surface Hierarchy

Three backgrounds, one layering order

Three background values create a subtle elevation system. The layering must be consistent — bg/neutral/foreground always sits above bg/neutral/middleground, and bg/neutral/background is the base page layer.

Principles

Four rules for color use

01
Always use semantic tokens in components.
Referencing text/primary instead of color/10 means the entire UI can adapt from a single source. Never hardcode hex values in components.
02
Verify contrast before using secondary or tertiary text.
text/secondary and text/tertiary may not pass 4.5:1 on all surfaces.
03
Reserve Apollo Yellow for brand moments.
Apollo Yellow is not a general-purpose highlight color. Do not use it as a status color, hover state, or interactive affordance.
04
Status colors are intentionally out of palette.
Pass green, advisory amber, and fail red carry a universal signal that must stay distinct from neutral surfaces.
